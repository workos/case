## Problem Statement

The Case harness pipeline was compared against Anthropic's "Harness Design for Long-Running Apps" article by three independent analyses (Claude, Codex, and synthesis). Case's foundations are strong — separate evaluator agents, fresh-context verification, file-based handoffs, deterministic orchestration, mechanical evidence gates. But five gaps emerged between Case's linear pipeline and the article's GAN-inspired iterative architecture:

1. **Under-planning normal issue work.** Ideation-sourced tasks get full contracts and specs, but issue-driven tasks go straight to implementation with only acceptance criteria. The implementer starts with fuzzy goals; the verifier checks against ad-hoc judgment rather than pre-agreed criteria.

2. **Under-structuring evaluator judgment.** The verifier and reviewer use prose-based prompts. There are no explicit rubric categories, no binary pass/fail per dimension, and no structured scoring in AGENT_RESULT. The article found untuned evaluators "talk themselves into approving."

3. **No harness-native revision loop.** When the verifier finds fixable issues, Case aborts (unattended) or asks the human (attended). The article runs 5-15 generator/evaluator cycles. Case has the retry branches but not a true evaluator-driven iteration loop with structured revision artifacts.

4. **One-size-fits-all pipeline.** A docs typo fix runs the same implement→verify→review→close chain as an ambiguous multi-file feature. The article argues evaluator overhead is only worth it when the task exceeds the model's solo capability boundary.

5. **Insufficient metrics for harness iteration.** Case tracks phase timings, CI first-push, and review findings, but can't answer: "Is the reviewer still worth it?" or "Does planning reduce revision loops?" No false-positive/negative rates, no task classification, no revision loop counts.

## Goals

1. **Sharpen implementation targets** — Every non-trivial pipeline run starts with explicit verification scenarios, non-goals, edge cases, and evidence expectations in the task file, so implementer and verifier share the same definition of "done"
2. **Structure evaluator judgment** — Verifier and reviewer use binary rubric categories with pass/fail verdicts and structured JSON in AGENT_RESULT, replacing prose-only evaluation
3. **Enable harness-native iteration** — When evaluators find fixable issues, the pipeline automatically feeds structured feedback back to the implementer without human mediation, capped at bounded cycles
4. **Right-size the pipeline** — Pipeline profile (tiny/standard/complex) is selected per task, so simple changes skip unnecessary phases and complex changes get planning
5. **Measure harness effectiveness** — Expanded metrics enable data-driven decisions about which pipeline components are still load-bearing as models improve

## Success Criteria

- [ ] Orchestrator generates done contract sections (Verification Scenarios, Non-Goals, Edge Cases, Evidence Expectations) in task .md for non-trivial tasks
- [ ] Ideation-sourced tasks with `contractPath` skip done contract generation (ideation contract subsumes it)
- [ ] Verifier AGENT_RESULT includes structured rubric with per-category pass/fail verdicts
- [ ] Reviewer AGENT_RESULT includes structured rubric with per-category pass/fail verdicts
- [ ] Pipeline automatically routes verifier/reviewer feedback back to implementer when findings are fixable (non-critical-structural)
- [ ] Revision loop is capped: max 2 verify cycles, each with 1 intelligent retry (worst case: 4 implementer spawns, 3 verifier spawns)
- [ ] Revision loop works in both attended and unattended modes
- [ ] `task.json` has a `profile` field (`tiny | standard | complex`) set by orchestrator at task creation
- [ ] Pipeline reads profile to determine phase list (tiny skips verify, complex adds planning)
- [ ] Run metrics track: task profile, revision loop count, human override count, evaluator false-positive/negative rates
- [ ] Retrospective learns from revision loops as a new signal type
- [ ] All existing tests pass; no breaking changes to task file format or .task.json schema (profile field is additive)

## Scope Boundaries

### In Scope

- Done contract sections in task .md (generated by orchestrator)
- Binary rubric categories for verifier and reviewer
- Structured revision request artifact and feedback loop in pipeline.ts
- `profile` field in TaskJson and profile-based phase selection
- Expanded MetricsCollector and RunMetrics types
- Updated verifier.md and reviewer.md prompts
- Updated assembler.ts to pass revision context
- Retrospective signal type for revision loops

### Out of Scope

- Planner as a separate agent — done contract generation happens inside orchestrator, not a new agent spawn
- Numeric scoring (1-5) for rubrics — binary pass/fail per category is sufficient; false precision is worse than honest binary
- Merging reviewer into verifier — keep them separate until metrics prove otherwise
- Longer-lived implementer sessions — experimental, needs A/B testing, defer until revision loop is proven
- Auto-profile selection from issue labels — orchestrator classifies via LLM judgment first; label-based heuristics are a future optimization
- Changes to ideation contract or spec templates

### Future Considerations

- Label-based or file-pattern-based automatic profile classification
- A/B testing simplified pipelines (e.g., skip reviewer for standard profile)
- Longer-lived implementer sessions for complex tasks
- Per-repo rubric customization (different rubric weights per repo type)
- Cost/token tracking per phase for ablation analysis
