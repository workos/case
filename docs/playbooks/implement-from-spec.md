# Playbook: Implement from Ideation Spec

Step-by-step guide for implementing a phase from an ideation specification. Use this when the orchestrator passes a **spec file path** instead of an issue summary.

## When to Use

- Task has `issueType: "ideation"` in the task JSON
- Orchestrator provides a `spec_path` instead of an issue summary
- The spec file was generated by the ideation plugin

## Step 1: Read the Spec

Read the spec file in full. If it references a template (`spec-template-*.md`), read that too.

Extract and retain:

| Section | What to use it for |
|---|---|
| Technical Approach | Overall strategy — understand before coding |
| Feedback Strategy | Inner-loop command and playground type — set up before coding |
| File Changes | New files, modified files — your scope of work |
| Implementation Details | Per-component instructions with code patterns — your implementation guide |
| Per-component Feedback Loops | Playground, experiment, check command — your inner loop per component |
| Testing Requirements | What tests to write |
| Validation Commands | What to run at the end (typecheck, lint, test, build) |

**The spec is your authority.** Follow its implementation details literally. Don't improvise beyond what's specified.

## Step 2: Set Up Feedback Environment

Before writing any code, establish the spec's feedback environment. This is a one-time setup.

1. Read the **Feedback Strategy** section — identify the inner-loop command and playground type
2. Verify the inner-loop command runs (even if it does nothing yet):
   ```bash
   {inner-loop command}  # e.g., pnpm test -- --filter bookmarks
   ```
3. Set up the playground if needed:
   - **Test runner**: Verify the test command works
   - **Dev server**: Start it and verify it's running
   - **Storybook/harness**: Start if the spec calls for one
4. If no Feedback Strategy section exists, fall back to the Validation Commands as post-implementation checks

## Step 3: Implement Components

For each component in the spec's **Implementation Details** section:

### With feedback loop (most components)

1. Read the **"Pattern to follow"** file if specified — match its conventions
2. **Create the playground artifact first** — test file with describe block, script harness, component stub
3. **Run the check command** to verify the loop works (expect trivial pass or "not implemented")
4. **Build incrementally**:
   - Implement one meaningful chunk (one function, one handler, one subcomponent)
   - Run the component's **check command** (or the spec's inner-loop command)
   - If failing: read output, fix, re-run. Iterate until passing.
   - Move to next chunk. Repeat.
5. **Run the experiment** — the parameterized edge-case check from the spec

### Without feedback loop (config, types, constants)

1. Implement fully per the spec's instructions
2. Run the spec's inner-loop command or validation commands to verify

## Step 4: Validate

Run all commands from the spec's **Validation Commands** section:

```bash
{typecheck command}   # e.g., pnpm typecheck
{test command}        # e.g., pnpm test
{lint command}        # e.g., pnpm lint
{build command}       # e.g., pnpm build
```

Fix any failures before proceeding.

## Step 5: Record & Commit

1. Pipe test output through `mark-tested.sh`:
   ```bash
   {test command} 2>&1 | bash /Users/nicknisi/Developer/case/scripts/mark-tested.sh
   ```
2. Commit with a conventional message:
   ```
   feat(scope): implement {phase description}
   ```
3. Append to the task file's Progress Log:
   ```markdown
   ### Implementer — {timestamp}
   - Spec: {spec-file-path}
   - Components: {list of components implemented}
   - Files changed: {count} ({list})
   - Tests: {pass count} passing
   - Commit: {hash}
   ```
4. Update task JSON agent phase to completed

## Multi-Phase Context

When implementing one phase of a multi-phase contract:

- Previous phases have already been committed to this branch
- Your changes build on top of their work
- The orchestrator runs a final full validation after all phases — your per-phase validation is for fast feedback, not the final gate
- Focus only on files and components listed in YOUR spec

## Common Pitfalls

- **Skipping feedback loop setup.** Set up before coding, not after. The loop catches errors immediately.
- **Ignoring "Pattern to follow" references.** The spec points to existing code for a reason — match its style.
- **Running the global test suite instead of scoped check command.** Use the fast inner loop during development. Save the full suite for Step 4.
- **Not reading existing files before modifying them.** Always read before edit.
- **Improvising beyond the spec.** If the spec doesn't mention it, don't add it.
- **Duplicating existing utilities.** Before creating a helper function (e.g., `generatePassword`, `formatDate`), search the codebase for existing implementations. Extracting a shared utility is better than having two copies diverge.
- **Missing test coverage for modified files.** If you add functionality to a file that has a companion `.spec.ts`, add tests for your changes there too — not just in the new spec files for new modules.
